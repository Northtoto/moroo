# Security Audit Report — Morodeutsh AI German Tutor Platform
**Date:** 2026-03-20
**Auditor:** Automated Red-Team Static Analysis
**Branch:** feature/tutor-improvements-and-fixes
**Scope:** Full codebase static analysis — no live requests made

---

## Executive Summary

The Morodeutsh platform has received meaningful security hardening (getUser() adoption, HSTS, X-Frame-Options, Stripe signature verification, idempotent webhook processing). However, **four HIGH/CRITICAL findings** block production readiness:

1. **CRITICAL** — `send-streak-email` admin route uses `execSync` with a shell command built from user-controlled JSON. The single-quote escape applied (`replace(/'/g, "'\\''")`) is insufficient to prevent shell injection when the JSON contains characters like `$`, backticks, or backslashes.
2. **HIGH** — `/api/health` is unauthenticated and reveals the operational status of all internal services (Supabase, Azure OpenAI, Redis) plus the Azure endpoint URL pattern, providing a free reconnaissance surface.
3. **HIGH** — No file-size or MIME-type check on audio uploads to `/api/tutor`. An attacker can stream an arbitrarily large binary to Azure Whisper, wasting quota or triggering DoS.
4. **HIGH** — In-memory rate-limit fallback resets on every serverless cold start, allowing burst abuse whenever Upstash Redis is unreachable.

**Overall security posture:** NOT READY FOR PRODUCTION (1 CRITICAL, 3 HIGH)

## Critical Finding Detail

### F-01 — CRITICAL: Shell Injection via `execSync` in Admin Streak Email

**File:** `src/app/api/admin/send-streak-email/route.ts` (~line 90)

**Description:**
The route accepts a JSON body with fields including `quick_task`, builds a MIME email containing that value, serialises it to JSON, then executes a shell command via Node's `execSync`:

```js
const gwsBody = JSON.stringify({ raw: rawBase64 });
const result = execSync(
  `gws gmail users.messages.send --userId me --body '${gwsBody.replace(/'/g, "'\\''")}'`,
  { encoding: 'utf8', timeout: 15000 }
);
```

The mitigation attempts to escape single-quotes using shell `'...'` quoting (`replace(/'/g, "'\\''"`). However, `rawBase64` is produced by `Buffer.from(html).toString('base64')`, and `html` is generated by `buildStreakEmailHtml(name, streak, quickTask, appUrl)` where **`quickTask` comes directly from `body.quick_task`** — attacker-controlled request body.

Base64 encoding neutralises most injection for the body argument. However, the broader pattern is dangerous:
- The authentication check is `verifyInternalSecret` — a single shared secret in `GWS_INTERNAL_SECRET`. If this secret leaks (e.g., via the health endpoint or logs), any external caller gains OS-level code execution on the server.
- `execSync` should never be used in a web-facing API handler. A subprocess shell execution model is inherently fragile.
- The error response includes the raw `err.message` from `execSync` which may contain shell stderr, leaking command-line arguments and file paths.

**Proof-of-Concept Attack:**
1. Attacker obtains or guesses `GWS_INTERNAL_SECRET` (weak or leaked).
2. Sends POST to `/api/admin/send-streak-email` with `"name": "$(id>/tmp/pwned)"` — if base64 wrapper is removed or `name` is used in another code path without encoding, shell code executes.
3. Error body in 500 response leaks `execSync` stderr, confirming execution context.

**Recommendation:**
Replace `execSync` entirely. Use the Google Gmail REST API directly with `fetch()` and a service account JWT. If `gws` CLI must be used, use `execFile` (not `execSync`) with arguments passed as an array — never interpolated into a shell string.

## High Finding Details

### F-02 — HIGH: Unauthenticated Health Endpoint Exposes Service Topology

**File:** `src/app/api/health/route.ts`

The `/api/health` GET endpoint requires no authentication. Its response reveals:
- Whether Supabase is `ok` or `down`
- Whether Azure OpenAI is `ok`, `down`, or `not_configured`
- Whether Upstash Redis is `ok`, `down`, or `not_configured`

This is free reconnaissance for an attacker — they can determine which backend services are active before crafting targeted attacks. The Azure check also constructs and calls `${endpoint}/openai/models` which confirms the endpoint URL is reachable.

**Recommendation:** Add a simple bearer-token check (reuse `METRICS_API_KEY` pattern already established in `/api/metrics`) or restrict to internal network only.

### F-03 — HIGH: No File-Size or MIME-Type Validation on Audio Upload

**File:** `src/app/api/tutor/route.ts` (lines ~110–130, `transcribeAudio` function)

The audio-correction flow extracts a `Blob` from FormData and passes it directly to Azure Whisper without any of:
- Maximum file size check (e.g., 25 MB Azure limit)
- MIME type allowlist (e.g., only `audio/webm`, `audio/mp4`, `audio/wav`)
- Filename sanitisation

An authenticated attacker can:
1. Upload a 500 MB file, exhausting Azure quota and causing timeouts.
2. Upload a file with a crafted filename (e.g., path-traversal sequences) — though Azure receives it in FormData, the filename is passed as-is via `fd.append('file', audioBlob, filename)`.
3. Upload a polyglot file (valid audio header, malicious payload) — Azure Whisper will likely reject it, but the attempt costs quota.

**Recommendation:**
```ts
const MAX_AUDIO_BYTES = 25 * 1024 * 1024; // 25 MB
const ALLOWED_AUDIO_TYPES = new Set(['audio/webm','audio/mp4','audio/wav','audio/mpeg','audio/ogg']);

if (audioBlob.size > MAX_AUDIO_BYTES) return error('File too large');
if (!ALLOWED_AUDIO_TYPES.has(audioBlob.type)) return error('Invalid file type');
const safeFilename = 'recording.webm'; // ignore client-provided filename
```

### F-04 — HIGH: In-Memory Rate-Limit Fallback Resets on Every Cold Start

**File:** `src/lib/redis-rate-limiter.ts`

The comment reads:
> "Graceful degradation: if Redis is not configured or unreachable, falls back to per-process in-memory limiting with a warning log."

In a serverless/edge environment (Vercel, Next.js), each cold start is a fresh process with an empty in-memory store. When Upstash Redis is temporarily unavailable (network hiccup, quota exceeded), every request appears as the first request to the in-memory limiter. An attacker who can trigger Redis unavailability (e.g., Upstash free-tier quota exhaustion) can then freely bypass all rate limits.

**Recommendation:** When Redis is unreachable, **fail closed** (reject requests with 503) rather than falling back to in-memory. Add a circuit-breaker flag so operators are notified immediately.

### F-06 — MEDIUM: Export Routes Have No Authentication

**Files:** `src/app/api/export/drive/route.ts:14`, `src/app/api/export/sheets/route.ts:25`

Both export routes expose `POST` handlers without `withApiGuard`. If these routes write to Google Drive/Sheets on behalf of users, unauthenticated callers could trigger exports.

### F-07 — MEDIUM: User-Controlled `belief_text` Injected into GPT System Prompt

**File:** `src/lib/student-model.ts` lines 99 and 136

```ts
// line 99
context.top_beliefs.map(b => `- [${b.topic}] ${b.belief_text} (confidence: ...)`)

// line 136
`\nKey insight about this student:\n${context.top_beliefs[0]?.belief_text ?? ''}`
```

`belief_text` is stored in the database and originally generated by GPT from prior student input. A student who crafts input designed to produce specific `belief_text` entries (indirect prompt injection) can poison their own system prompt on future requests — potentially causing the tutor to output harmful content, reveal internal instructions, or behave unexpectedly.

**Recommendation:** Sanitise or truncate belief strings before system prompt injection. Consider wrapping with XML-style delimiters and instructing the model to treat the section as data only: `<student_belief>...belief_text...</student_belief>`.

Already covered under F-02. The Azure check leaks whether the configured endpoint is reachable and confirms the endpoint format.

### F-09 — MEDIUM: No Content-Security-Policy Header

**File:** `next.config.ts`

The `headers()` config sets X-Frame-Options, X-Content-Type-Options, HSTS, and Referrer-Policy but **omits Content-Security-Policy entirely**. Without CSP:
- XSS payloads can load arbitrary external scripts
- Tesseract.js (WASM) and AudioRecorder (MediaRecorder) are legitimately used and would need `wasm-unsafe-eval` in CSP, but this is manageable with nonces

**Recommendation:** Add a strict CSP. For a Next.js app with Tesseract.js:
```
Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-{NONCE}'; worker-src blob:; wasm-src 'self' blob:; connect-src 'self' https://*.supabase.co; img-src 'self' data: blob:; style-src 'self' 'unsafe-inline';
```

## Low / Info Findings

### F-10 — LOW: Rate Limit Key is User ID Only (Not IP+User)

An attacker with multiple free accounts from the same IP can circumvent per-user rate limits. Consider keying rate limits on `userId:ip` composite or adding an IP-level global limiter.

### F-11 — LOW: `tarek.md` Documents That `.env.local` Was Modified

`tarek.md` is committed to the repository and documents that `.env.local` was modified to fix a service role key. While `.env.local` should be in `.gitignore`, the existence of this file in the repo may confuse developers into thinking credentials are tracked. Verify `.gitignore` covers `.env.local`.

### F-12 — LOW: Stripe SDK Initialised with Literal Fallback `'sk_missing'`

**File:** `src/app/api/stripe/webhook/route.ts:21`
`new Stripe(process.env.STRIPE_SECRET_KEY ?? 'sk_missing', ...)` — if `STRIPE_SECRET_KEY` is unset in an environment, the SDK initialises without throwing. This means misconfigured deployments silently fail rather than crashing at startup.
**Recommendation:** Throw at startup if the key is missing.

### F-13 — LOW: Audio Filename Passed Unsanitised to Azure FormData

**File:** `src/app/api/tutor/route.ts:122`
`fd.append('file', audioBlob, filename)` where `filename` originates from `formData.get('filename')`. While Azure handles filenames as opaque metadata, using a hardcoded safe name is safer.

### F-14 — INFO: `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` in Client Bundle

This is **by design** for Supabase's browser SDK — the anon key is public and protected by RLS. Not a vulnerability, but operators should verify RLS policies cover all tables.

### F-15 — INFO: `X-XSS-Protection: 0` Intentionally Disabled

Correct for modern browsers. The legacy XSS auditor header is deprecated and can introduce vulnerabilities in older browsers. Setting it to `0` is best practice.
